## Трекер задач сотрудников

### Что исправлено (добавлено поверх предыдущей ветки employees):
1) Исправлена межполевная валидация `EmployeeSerializer.validate()` для `PATCH`

####  Проблема: 
В `validate()` раньше использовалось:
- `email = attrs.get("email")`
- `is_active = attrs.get("is_active", True)`

При `PATCH` в `attrs` приходят только изменяемые поля. Если обновлять только position, то:
- `email` становится `None`
- `is_active` становился `True` по умолчанию

В результате сериализатор ошибочно возвращал 400, даже если email уже есть в базе.

#### Решение:
Учесть режим обновления и использовать `self.instance`:
- если поле не передано в запросе, берём его значение из базы
- для `POST` оставляем логику по умолчанию (`is_active=True`)

Это делает валидацию корректной для:
- `POST` (создание)
- `PUT` (полное обновление)
- `PATCH` (частичное обновление)

2) Расширен поиск по сотрудникам
Поиск через ?search= можно расширить до:
- `full_name` (ФИО)
- `position` (должность)
- `email` (почта)

Это улучшает удобство работы с API при тестировании и для будущего фронтенда.

3) `HealthCheckView` оставлен минимальным
`request` может не использоваться и это нормально, сигнатура метода стандартная для DRF.

## Ветка `feature/api-crud-tasks`

### Реализованная функциональность:

#### 1. CRUD API для задач (Task)

Реализованы стандартные REST-операции:

| Метод  | URL                | Описание                    |
| ------ | ------------------ | --------------------------- |
| GET    | `/api/tasks/`      | Получение списка задач      |
| POST   | `/api/tasks/`      | Создание новой задачи       |
| GET    | `/api/tasks/{id}/` | Получение задачи по id      |
| PATCH  | `/api/tasks/{id}/` | Частичное обновление задачи |
| DELETE | `/api/tasks/{id}/` | Удаление задачи             |

#### 2. Поддерживаемые поля задачи
- `title` - название задачи
- `description` - описание
- `assignee` - исполнитель (FK → Employee, передаётся как id)
- `owner` - владелец задачи (FK → Employee)
- `status` - статус задачи (NEW, IN_PROGRESS, REVIEW, DONE)
- `due_date` - срок выполнения
- `report_file` - файл отчёта 
- `review_comment` - комментарий проверяющего
- `created_at` - дата создания (`read-only`)

#### В ответах API дополнительно возвращаются:
- `assignee_full_name`
- `owner_full_name`

#### 3. Валидации и бизнес-правила
На уровне модели `Task.clean()`:
- отчёт (`report_file`) разрешён только для статусов DONE и REVIEW
- для статуса DONE отчёт обязателен
- владелец задачи (`owner`) не может быть её исполнителем (`assignee`)
- валидация выполняется при каждом `save()` через `full_clean()`

На уровне сериализатора:
- валидация даты (`due_date` не может быть в прошлом)
- корректная работа для POST, PUT, PATCH
- дружелюбные сообщения об ошибках API

#### 4. Фильтрация, поиск и сортировка

Для списка задач доступны:
- фильтрация по:
    - `status` 
    - `assignee `
    - `owner`
  
- поиск по:
   - `title `
   - `description`
  
- сортировка по:
    - `created_at `
    - `due_date`
    - `status`

Примеры запросов:
```
GET /api/tasks/
GET /api/tasks/?status=IN_PROGRESS
POST /api/tasks/
PATCH /api/tasks/{id}/
```
